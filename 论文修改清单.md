## 2. 第三章 系统需求分析 →（二）功能需求分析  [更新+新增]

原文 8 条功能模块，按下列调整（改动处已加粗）：

> 1. **客户管理模块**。记录客户姓名、联系方式、**所在村**、详细地址（选填）和备注，支持新增、修改、查询及查看该客户关联的历史订单；客户的村从村字典中选择，便于后续服务费匹配。
> 2. **宴席订单管理模块**。维护宴席类型、举办日期、**村与详细地点**、桌数、人数、预算、**服务费**、状态及特殊要求等信息，是系统的核心主线。
> 5. **物料与人工管理模块**。记录食材、餐具、车辆等物料（含**库存数量**）及厨师、帮厨、服务员的派工；并支持在宴席执行前登记**订单需准备的物料及数量**。
> 6. **执行与结算模块**。记录宴席执行情况、变更记录、实际费用、收款与结算结果；结算时对准备物料进行**清点核对并自动计算损耗**。
> 7. **统计分析模块**。按日期、宴席类型、营收、成本、**服务费收入、物料损耗**等维度生成统计，支持经营复盘。

新增一条功能模块：

> 9. **价格带（服务费定价）模块**。按「**村 / 日期区间 / 桌数范围**」维护不同的整单服务费规则；新建订单时系统据订单的村、举办日期与桌数**自动匹配服务费并带入订单（可手动调整）**，匹配遵循「优先级高者优先、条件更具体者优先」，并设默认兜底价。村字典与价格带规则的维护由管理员负责（新增村对所有登录用户开放）。

## 3. 第四章 数据库设计（改动最大）

### 3.1（二）概念结构设计 · 表4.1 核心实体表  [更新+新增，整表替换]

> 本系统通过实体-联系（E-R）方法对业务对象进行建模，识别出 **15 个核心实体**：

| 实体名称 | 业务含义 | 关键属性 |
|---|---|---|
| 用户（User） | 系统登录账号 | 账号、密码、角色、联系方式 |
| 村（Village）**新增** | 地点字典，按村归类 | 村名 |
| 客户（Customer） | 宴席的需求方 | 姓名、电话、**所在村**、详细地址、备注 |
| 宴席订单（BanquetOrder） | 一次宴席承办合同 | 订单号、类型、日期、**村**、地点、桌数、人数、预算、**服务费**、状态 |
| 菜单模板（Menu） | 可复用的套餐组合 | 名称、适用场景、参考价格 |
| 菜品（Dish） | 独立的菜品单元 | 名称、类别、单价 |
| 菜单菜品明细（MenuDish）**新增** | 菜单模板与菜品的固定配比 | 数量、排序、备注 |
| 订单菜单明细（OrderMenuDetail） | 订单与菜单/菜品的关联记录 | 数量、备注 |
| 物料（Material） | 食材、餐具、车辆等执行资源 | 名称、类别、单位、单价、**库存** |
| 订单准备物料（OrderMaterial）**新增** | 某订单需准备的物料及数量 | 准备数、清点结果、单价快照 |
| 物料损耗（MaterialLoss）**新增** | 结算清点后产生的损耗台账 | 准备数、清点数、损耗数、单价、损耗金额 |
| 人工安排（LaborArrangement） | 厨师、帮厨、服务员、司机的派工 | 角色、姓名、电话、费用 |
| 价格带（PriceBand）**新增** | 按村/日期/桌数设定整单服务费 | 名称、村、日期区间、桌数范围、服务费、优先级、启用 |
| 变更记录（ChangeRecord） | 订单执行过程中的每一次调整 | 类型（改期/加桌/退订）、变更前值、变更后值、原因、操作者 |
| 结算（Settlement） | 宴席完成后的财务结算 | 总金额、成本、实收、支付方式 |

### 3.2 实体联系补充  [新增，接在原 1:N / 1:1 描述之后]

> - **村 — 客户 / 宴席订单 / 价格带（1:N）**：一个村可对应多个客户、多个订单、多条价格带规则；删除村时相关外键置空（ON DELETE SET NULL），不影响历史数据。
> - **菜单模板 — 菜单菜品明细 — 菜品（M:N）**：以 `menu_dishes` 作为中间表固化"某套餐包含哪些菜、各多少份"，下单选用模板时据此自动展开为多条订单菜品明细。
> - **宴席订单 — 订单准备物料（1:N）**：一个订单可登记多条需准备的物料。
> - **宴席订单 — 物料损耗（1:N）**：结算清点后，一个订单可产生多条物料损耗记录（按 `(order_id, material_id)` 唯一）。

### 3.3（三）逻辑结构设计  [更新+新增，整段替换]

> 系统采用关系型数据库模型，共设计 **15 张主要数据表**：

> 1. `users` 用户表：登录账号、密码、角色、联系方式。
> 2. `villages` **村字典表（新增）**：村名（唯一）。
> 3. `customers` 客户表：姓名、联系方式、**所在村（village_id）**、详细地址、备注。
> 4. `banquet_orders` 宴席订单表：订单号、客户编号、宴席类型、举办日期、**村（village_id）**、地点、桌数、人数、预算、**服务费（service_fee）**、状态、备注。
> 5. `menus` 菜单表：名称、适用场景、价格、说明。
> 6. `dishes` 菜品表：名称、类别、单价、说明。
> 7. `menu_dishes` **菜单菜品明细表（新增）**：菜单与菜品的固定配比（数量、排序），按 `(menu_id, dish_id)` 唯一。
> 8. `order_menu_details` 订单菜单明细表：订单与菜单、菜品的对应关系。
> 9. `materials` 物料表：名称、类别、单位、单价、**库存（stock）**。
> 10. `labor_arrangements` 人工安排表：角色、姓名、电话、费用。
> 11. `price_bands` **价格带表（新增）**：规则名、村、起止日期、桌数下限/上限、整单服务费、优先级、是否启用。
> 12. `change_records` 变更记录表：保存**改期、加桌、退订**等变更操作的前后值、原因、操作者与时间。
> 13. `settlements` 结算表：订单总金额、成本、实收金额、支付方式、结算时间。
> 14. `order_materials` **订单准备物料表（新增）**：订单编号、物料编号、准备数量、清点结果、单价快照。
> 15. `material_losses` **物料损耗表（新增）**：订单编号、物料编号及名称、准备数、清点数、损耗数、单价、损耗金额。

> 说明：`change_records` 的变更类型由原来的「改期/加桌/换菜/退订」**调整为「改期/加桌/退订」**——换菜已从变更记录中移除，菜品增减统一在菜单明细中维护。

### 3.4（四）索引设计  [更新：原文 5 条索引现已真正落到 schema，并补充新表索引]

> 注：论文原文列出的 5 条索引此前**并未写进建表 SQL**，现已补齐；并按订单详情页"按 order_id 并发加载各明细"与"按村匹配/统计"等场景增加索引。完整索引如下（主键与 UNIQUE 约束已自动建索引，此处为补充的业务查询索引）：

```sql
CREATE INDEX idx_orders_event_date     ON banquet_orders(event_date);   -- 按日期筛选 / 档期冲突检测
CREATE INDEX idx_orders_status         ON banquet_orders(status);       -- 状态过滤
CREATE INDEX idx_orders_customer       ON banquet_orders(customer_id);  -- 客户历史订单
CREATE INDEX idx_orders_village        ON banquet_orders(village_id);   -- 按村匹配/统计
CREATE INDEX idx_customers_phone       ON customers(phone);             -- 客户电话搜索
CREATE INDEX idx_customers_village     ON customers(village_id);        -- 按村筛选客户
CREATE INDEX idx_order_menu_order      ON order_menu_details(order_id); -- 详情页加载菜单明细
CREATE INDEX idx_labor_order           ON labor_arrangements(order_id); -- 详情页加载人工
CREATE INDEX idx_changes_order         ON change_records(order_id);     -- 详情页加载变更记录
CREATE INDEX idx_order_materials_order ON order_materials(order_id);    -- 详情页加载准备物料
```

> `menu_dishes(menu_id,dish_id)`、`material_losses(order_id,material_id)`、`settlements(order_id)` 已由 **UNIQUE 约束**自动建立（复合）索引，按 `order_id`/`menu_id` 前缀查询即可命中，无需重复创建；`price_bands` 表数据量很小，全表扫描即可。

### 3.5（五）典型查询优化示例  [更新]

- **订单详情聚合查询** —— 原文"返回 **5 类数据**、Promise.all 并发 **5 条 SQL**"改为：

> 订单详情页一次性返回订单基础信息及 **6 类明细数据：菜单明细、人工安排、变更记录、结算、准备物料、物料损耗**。后端采用 `Promise.all` 并发执行 6 条独立 SQL，均基于外键 `order_id` 的索引查找，相较串行执行可显著压缩响应时间。

- **统计分析的条件聚合** —— 在原 SQL 基础上增加服务费收入聚合：

```sql
SELECT COUNT(*) AS total_orders,
       COUNT(*) FILTER (WHERE status='completed') AS completed_orders,
       COUNT(*) FILTER (WHERE status='cancelled') AS cancelled_orders,
       COALESCE(SUM(s.total_amount), 0) AS total_revenue,
       COALESCE(SUM(s.actual_cost), 0)  AS total_cost,
       COALESCE(SUM(o.service_fee) FILTER (WHERE o.status='completed'), 0) AS total_service_fee
FROM banquet_orders o
LEFT JOIN settlements s ON o.id = s.order_id
WHERE o.event_date BETWEEN $1 AND $2;
```

- **价格带匹配（新增示例）**：

> 系统取出全部启用的价格带规则，在应用层按"村相等或不限 ∧ 举办日期落在区间或不限 ∧ `min_tables ≤ 桌数 < max_tables`（左闭右开）"过滤，再按「优先级降序 → 条件具体度降序 → id 降序」排序取首条，得到该订单的整单服务费；无任何专属规则命中时落到默认兜底价。

## 4. 第五章 系统详细设计与实现

### 4.1（一）客户管理模块  [更新+新增]

- 列表列与表单字段更新：

> 客户编辑表单包含姓名、联系电话、**村**、**详细地址（选填）**与备注，其中姓名与电话为必填项；"村"为下拉选择，下拉旁提供"**+ 新增村**"按钮，可在不离开表单的情况下就地新增村并自动选中。客户列表的"村 / 地址"列优先显示所在村。

- 新增子页（建议加一小节并配图）：

> 客户管理页拆分为"客户"与"价格带"两个标签（价格带标签仅管理员可见）。价格带页上半部分为**村字典**（卡片式网格，支持搜索、行内重命名与删除）；下半部分为**价格带规则表**，可按村、日期区间、桌数范围、服务费、优先级与启用状态维护规则。新增订单时即据此自动匹配服务费。

### 4.2（二）宴席订单管理模块  [更新+新增]

- 下单字段：

> 创建订单时需填写宴席类型、执行日期、**村**、详细地点、桌数、人数、预算等信息；其中**服务费由系统按价格带自动匹配并带入（可手动调整）**，仅在填写桌数后匹配，以避免落到误导性的兜底价。关联已有客户时会自动带出该客户的村与地址。

- 订单详情整合区：

> 订单详情页将基本信息、菜单明细、**准备物料**、人工安排、变更记录与结算情况整合到同一页面。

- 新增"已完成订单锁定"（建议加一段）：

> 为保证已结算订单的数据稳定，订单进入"已完成"状态后，系统**锁定其菜单、准备物料、人工与结算的修改入口**：前端将相关按钮与输入置灰/隐藏，后端对相应写接口统一返回拒绝（HTTP 403）。如确需调整，须先将订单状态改回非完成态再操作。

- 变更记录说明改写（原文"保留变更前后取值与操作人"基础上补充其"即时生效"语义）：

> 变更记录采用"**记录即生效**"机制：改期、加桌、退订三类变更在写入记录的同时，由服务端依订单当前值推导"变更前"并**直接回写订单对应字段**（举办日期 / 桌数 / 状态），并做合法性校验（日期格式、桌数不小于 1 等）。菜品增减在菜单页单独维护，不再作为变更类型。

### 4.3（四）菜单管理模块  [更新]

> 通过菜单菜品明细（`menu_dishes`）固化每个套餐包含的菜品及份数；下单选用菜单模板时，系统**自动将模板展开为多条订单菜品明细**，无需逐道重复录入，也可在此基础上灵活增减。

### 4.4（五）物料与人工管理模块  [新增，缺口最大]

> 本系统的物料以**可重复使用资源**为主（如桌子、椅子、碗筷、帐篷等），因此库存管理采用"**借出—归还**"模型而非"消耗"模型。物料管理在原有名称、类别、单位、单价基础上**新增库存数量**；物料详情可查看**该物料的库存当前被哪些订单占用（借出）**。
>
> 宴席执行前，可在订单的"**准备**"页登记需要准备的物料及数量，系统**按数量从库存借出（扣减），库存不足时拦截并提示**；移除某条准备记录会把仍在外的数量退还库存。这样把"借出了哪些物料、借出多少"纳入可控管理，并为结算阶段的清点归还与损耗核对提供依据。

### 4.5（六）结算与统计分析模块  [更新+新增]

- 结算财务模型（整段重写）：

> 结算页围绕"**服务费 → 实际收款 → 利润**"组织财务核对。展示与录入项包括：**服务费**（来自价格带，参考价）、**实际收款金额**（录入）、**额外损耗**（可选录入，如人工估损、临时报废等）；并实时派生：
> - **折扣 = 服务费 − 实际收款**（实收少于服务费时显示）；若实收多于服务费，则显示为**额外收入 = 实际收款 − 服务费**；
> - **物料损耗费**：由物料清点核对自动计算（见 4.4），逐项录入"清点结果（归还/完好数）"，归还数回补库存，损耗 = 准备数 − 清点数；
> - **损耗合计 = 物料损耗费 + 额外损耗**；
> - **利润 = 实际收款 − 损耗合计**。
>
> 此外，**帮厨费用**（人工费用合计）作为经营者需转付给帮厨的**转付项单独展示，不计入利润**。后端在保存结算时，按"营收 = 实际收款、成本（总损耗）= 物料损耗 + 额外损耗"派生存储 `total_amount` / `actual_cost`，以与统计模块口径保持一致。

- 统计补充：

> 统计分析在总订单数、按月趋势与宴席类型分布的基础上，**口径调整为：总营收（实收）、总损耗（物料损耗 + 额外损耗）、利润 = 营收 − 总损耗**；并提供**合同服务费合计**（参考项，不重复计入营收）与**物料损耗统计**（其中物料损耗按时间汇总数量与金额、并按物料给出损耗排行）。

## 5. 第六章 系统测试 →（三）测试用例设计  [新增用例]

在原 6 条用例后补充：

> 7. **价格带匹配测试**：在不同村、不同日期、不同桌数下新建订单，应自动带出对应价格带的服务费；命中多条时按优先级/具体度取值；无专属规则时取默认兜底价；手动改写后不再被自动覆盖。
> 8. **物料库存测试**：登记准备物料应按数量扣减库存；当准备数超过当前库存时应拦截并提示"库存不足"；移除准备物料应回退库存。
> 9. **损耗计算测试**：结算录入清点结果后，应按"准备数 − 清点数"正确生成损耗记录与金额，并在统计中累计；重复结算据最新清点重算而不累加。
> 10. **完成订单锁定测试**：订单置为"已完成"后，再尝试增删菜单/准备/人工或重复结算，应被拒绝并提示订单已完成。
> 11. **客户选村测试**：新建客户时选择村应正确保存并在列表显示；无村时可就地新增村并自动选中。

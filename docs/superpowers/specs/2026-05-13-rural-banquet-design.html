农村流水席信息管理系统

设计文档 · 2026-05-13
React 18 + Vite Tailwind CSS Express.js PostgreSQL JWT 认证 JavaScript
1 背景与目标

农村流水席（婚宴、寿宴、乔迁宴、满月宴）长期依赖电话、微信和纸笔管理。当业务量增加或多场宴席并行时，容易出现以下问题：

档期冲突
同天接单未发现时间重叠
变更遗漏
临时加桌、改期口头通知后遗忘
结算混乱
食材、人工、收款分散记录难对账
信息分散
关键信息在个人手机，他人难以接手

系统目标：把原来靠经验记住的规则固化到系统里，实现从 客户咨询 → 报价下单 → 档期确认 → 宴席执行 → 费用结算 → 统计分析 的全过程管理。

2 技术架构
前端框架 React 18 + Vite
前端样式 Tailwind CSS
后端框架 Express.js (Node.js)
数据库 PostgreSQL
认证方式 JWT（24h 有效期）
数据库驱动 pg（原生 SQL）
系统架构示意
浏览器（React + Tailwind CSS）
        ↕ HTTP / JSON
Express.js 后端（业务逻辑 · JWT认证 · 档期检测 · 结算计算）
        ↕ SQL
PostgreSQL（客户 · 订单 · 菜单 · 变更记录 · 结算数据）

Vite 开发服务器代理到后端 3001 端口；JWT 存于 localStorage，每次请求携带 Authorization 头。

4 数据库设计
4.1 表清单（共 10 张表）
表名用途
users系统用户（admin / staff）
customers客户基础信息（姓名、电话、地址、备注）
banquet_orders宴席订单 （核心表），连接所有其他表
menus菜单模板（婚宴套餐、寿宴套餐等）
dishes菜品（名称、类别、单价、说明）
order_menu_details订单-菜品关联明细（多对多中间表）
materials物料/餐具/车辆等资源信息
labor_arrangements人工安排（厨师、帮厨、服务员）
change_records变更记录（加桌、改期、换菜、退订）
settlements结算（总金额、成本、实收、未收）
4.2 订单状态流转
待确认 → 已确认 → 执行中 → 已完成
任意阶段均可 → 已取消
4.3 核心字段 — banquet_orders
CREATE TABLE banquet_orders ( id SERIAL PRIMARY KEY, order_no VARCHAR(20) UNIQUE NOT NULL, -- 如 BQ20260513001 customer_id INTEGER REFERENCES customers(id), banquet_type VARCHAR(20), -- 婚宴/寿宴/乔迁宴/满月宴 event_date DATE NOT NULL, location TEXT, table_count INTEGER, guest_count INTEGER, budget NUMERIC(10,2), status VARCHAR(20) DEFAULT 'pending', notes TEXT, created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW() );
4.4 核心字段 — settlements
CREATE TABLE settlements ( id SERIAL PRIMARY KEY, order_id INTEGER UNIQUE REFERENCES banquet_orders(id), total_amount NUMERIC(10,2), -- 订单总金额 actual_cost NUMERIC(10,2), -- 实际成本 received_amount NUMERIC(10,2), -- 已收款 -- 未收金额 = total_amount - received_amount（前端展示计算） payment_method VARCHAR(20), -- 现金/微信/转账 notes TEXT, settled_at TIMESTAMPTZ );

6 档期冲突检测逻辑

后端在 POST /api/orders 时执行以下步骤：

1
查询 banquet_orders 中 event_date = 新订单日期，且状态不为 cancelled
2
若当天已有 1–2 场宴席 → 返回 info 提示：「当天已有 N 场，请注意资源安排」
3
若当天已有 ≥ 3 场宴席 → 返回 warning 提示：「当天订单已满，是否确认继续接单？」
4
前端展示 ScheduleAlert 组件，用户确认后可强制保存，不强制阻止

9 非功能需求实现方式
易用性
表单字段名贴合业务习惯（"宴席类型"而非"type"），状态用中文标签展示
可靠性
必填字段后端校验，档期检测在服务端执行，不依赖前端 JS 拦截
安全性
JWT 过期时间 24h，密码 bcrypt 加密，删除操作有 admin 角色中间件保护
可维护性
路由按模块拆分，数据库操作集中在 routes 文件，便于后续扩展微信提醒等功能

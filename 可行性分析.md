# 二、可行性分析

在真正动手做需求分析、把系统开发出来之前，先对这个课题到底可不可行做一番交代是有必要的。说到底，可行性分析想弄清楚的就是几件事："农村流水席信息管理系统"这东西值不值得做、做不做得出来，以及真正做完之后能不能用得起来。把这几点先想明白，就不至于在方向还没理清、条件也还不够的情况下贸然动手，最终白白耗费时间和精力。本课题针对的是农村里规模不大的经营团队，重心又落在后台的业务管理上，因此下面就分技术、经济、操作这三块分别来看，再回过头对整体作出判断。

## （一）技术可行性

技术可行性要看的，是在眼下这些技术条件、再加上开发者手里已有的技术积累下，能不能照着设计目标把系统真正做出来。本系统走的是 B/S 也就是浏览器/服务器这一套架构，开发上采用前后端分离的方式，整条技术路线比较清楚，配套的生态也已经相当成熟，单从技术角度讲是没有问题的。

**1. 架构方案成熟可靠**

本系统在架构上选的是 B/S 加前后端分离这一组合：前端在浏览器里运行，负责管理操作；后端则以接口、也就是 API 的形式对外提供数据，两边靠 HTTP 协议来回沟通。这样划分之后，界面、业务逻辑和数据存储就被分到了几个相对独立的层里，分工开发时各自独立，往后维护、扩展也都更为便利。前后端分离和分层这套做法，在餐饮类的信息管理系统里已经应用得比较广泛、也较为成熟[4][7]，本系统沿用它，工程上是有先例可循的，也谈不上有难以攻克的技术障碍。

**2. 后端技术成熟**

后端采用 Node.js 运行环境，并以 Express 框架搭建 Web 服务。Node.js 采用事件驱动、非阻塞 I/O 模型，对于本系统中表单提交、列表查询、状态更新这类出现频率较高、但单次开销不大的请求，处理起来较为合适；Express 框架本身轻量灵活，路由划分与接口开发都能较快完成。在用户身份验证方面，系统采用 JWT，即 JSON Web Token 来管理登录状态，用户口令则经 bcrypt 加密后存储，基本的安全性由此得到保障。上述技术均为开源领域的主流选择，社区资料丰富，出现问题时也便于排查，整体技术风险较低。

**3. 前端技术成熟**

前端采用 React 框架进行开发。React 以组件化方式组织页面，对于订单录入、订单列表、菜单维护、详情查看、统计展示这类结构较为规整的界面，用它来构建较为合适，组件复用程度也高，开发与后期维护的成本随之降低。React 生态比较完整，文档也较为清晰，能够较快地搭建出交互清晰、响应及时的管理界面。

**4. 前端构建工具成熟**

前端的构建与开发调试由 Vite 来承担。Vite 是新一代的前端构建工具：在开发阶段，它借助浏览器原生的 ES Module 机制，并通过 esbuild 对第三方依赖进行预构建，因此项目启动较快，源码改动后还能通过模块热替换（HMR）即时反映到页面上，调试体验较好；在打包阶段，它基于 Rollup 进行构建，能够输出经过优化、体积相对较小的静态资源。本系统通过 @vitejs/plugin-react 插件把 Vite 与 React 结合起来使用，整体配置简洁、上手成本不高。选用 Vite，一方面缩短了开发过程中的等待时间、提升了迭代效率，另一方面也让前端工程的组织更加规范，进一步保证了前端开发在技术上的可行性。

**5. 数据库技术成熟且部署灵活**

数据存储采用关系型数据库，而且系统把 PostgreSQL 和 SQLite 两套都兼容了下来：正式部署的环境里可以使用 PostgreSQL，它对关系数据、各种约束还有事务的支持都比较完整，客户、订单、菜单、物料、结算这些彼此关联很清楚的业务数据，交给它处理是合适的；要是只是轻量运行或者做演示，那就换成零配置的 SQLite，连数据库服务都不用单独安装就能运行。已有研究就专门针对网络不太稳的使用场景，设计过一套支持离线、又方便往后迭代维护的餐饮系统[11]，本系统这种双数据库加本地化部署的路子与之思路一致，放到网络和运维条件都比较有限的农村场景里，同样用得上。这种"双数据库"设计并没有让开发变得更麻烦，却把部署时的灵活性提了上去，技术上的把握自然也更足了。

**6. 开发者技术储备充足**

这个课题用到的 Node.js、Express、React，还有关系型数据库，在软件工程专业里都算常见的技术栈，开发者在平时上课和动手实践里，底子已经打得差不多了。再说这些技术的官方文档、开源的例子、社区里的讨论都不少，真碰上拿不准的地方，能查的资料也很多。这么看下来，不管是开发者自己的能力，还是外头能借到的支持，本系统在技术上要做出来都是可行的。

几方面摆在一起看，本系统挑的技术方案本身比较稳定、彼此之间也能配套，开发者手里又有相应的技术能力，**技术可行性成立**。

## （二）经济可行性

经济可行性要算的，无非是系统在开发、部署、运行这几步里要花多少成本，以及真正用起来之后又能带回多少好处，这一进一出划不划算，也就清楚了。对本系统这类面向农村小规模经营团队的项目而言，这笔账能不能算得过来，往往直接决定了它最后能不能真的落地。

**1. 开发成本低**

本系统从头到尾都是采用开源、不要授权费的技术搭起来的，Node.js、Express、React、PostgreSQL、SQLite 全都能免费使用，不存在商业软件那种买授权的开销。开发用的工具也一样，基本就是免费的代码编辑器加上开源组件，这么一来，软件这一层的开发成本几乎可以当成零，真正投入的，主要是开发者的时间和精力。况且这本就是毕业设计，开发由学生独立完成，额外的人力开销也省了。

**2. 硬件与部署成本低**

本系统就运行在普通计算机加浏览器的环境里，对硬件没什么高要求，那种贵价的服务器、专用设备也都不必去买。靠着 SQLite 不用配置这一特性，甚至一台普通电脑就能把部署和使用一并搞定；哪天业务做大了，再把数据库换成 PostgreSQL，核心那套逻辑也不用改。已有研究也是面向中小餐饮商家，给出过门槛低、操作简单、成本也比较可控的轻量化数字化方案[9]，本系统的定位跟它一致，农村这种小本经营场景对低成本的实在要求，正好能对上。

**3. 运维成本低**

系统的结构搭得比较清晰，模块之间怎么分也很明确，平时运维要操心的，无非是把数据备份做好、偶尔冒出来的小毛病随手处理掉，用不着专门安排一支技术队伍盯着。前后端分离这一设计也帮了忙，往后想调个功能、升级某一块，都更好动手，长远算下来，维护这块的花费又往下压了压。

**4. 效益分析**

本系统虽然不会直接挣来钱，可它在经济上的好处，体现在"经营效率往上走、潜在损失往下降"这两个方面。把订单、档期、菜单、物料、结算这些信息统一管起来之后，像档期撞车、临时改了又忘、事后对不上账这类暗里的损失，就能减少不少；物料备了多少、损耗了多少都有记录，食材的浪费也跟着好控制，成本自然省下来。在经营分析方面，已有研究专门做过统计模块，基于点餐、结算这些数据来为管理者提供决策依据[8]；也有研究指出，把财务和经营的数据整合到一块儿分析，决策能更靠谱、也更及时[12]。本系统的结算和统计功能走的正是这个路子，经营者想回头复盘、想找个参考再拿主意，都方便。在投入这么低的前提下，这些好处对小本经营的团队来说，是实实在在有用的。

算下来，本系统花得不多、回报又实在，**经济可行性成立**。

## （三）操作可行性

操作可行性盯着的，是系统真正用起来之后，那些目标用户愿不愿意用、用着顺不顺手。本系统要面对的用户，是农村流水席的经营者，还有跟他们一块儿干活的内部人手，这些人大多没受过什么专业的计算机训练，正因如此，系统好不好上手就格外要紧。

**1. 贴合真实业务流程**

本系统不是凭空设计出来的，而是顺着农村流水席"接单—安排—执行—结算"这条真实的流程来建的模，里头的功能模块，跟经营者平时干活的一个个环节是对得上号的。用户上手操作时，先做哪步后做哪步，跟他原来的习惯八九不离十，不必再去重新摸索一套陌生的业务逻辑，门槛也就跟着低了下来。

**2. 界面简单、学习成本低**

系统的定位就是个后台管理工具，界面来来去去也就是表单录入、列表查询、详情查看这几样，怎么点怎么用都比较直观，没有那种绕来绕去的操作流程。普通用户稍微带一带、培训一下就会用了，订单录入、看档期、记变更、做结算这些常干的活儿，都能自己独立搞定。已有面向多角色的餐厅管理系统在实践里已经说明，把经营者、操作人员这些不同角色拢到同一个系统里一起管，是行得通的[6]，本系统在这点上同样不难用。

**3. 角色权限清晰、协作顺畅**

系统里分了几种用户角色，像管理员、普通员工就是，关键的操作还另外卡了权限。这么安排，一头把结算信息这类核心数据的安全守住了，另一头又能让好几个协作的人各自在权限范围里分头干活。哪怕中途换了人、不是同一个人从头跟到尾，后来的人照着系统里留下的记录也能接着往下做，过去那种事情全集中在某一个人手里、别人难以接手的毛病，也就避开了。

**4. 用户接受度高**

经营者最头疼的，无非是"信息容易丢、容易乱、过后又难复盘"这几件事，系统恰恰把它们给解了，用过一阵子就会发现，翻来覆去找聊天记录、全凭记忆核对这种重复活儿明显少了，便利一旦摸得着，用户自然也就愿意一直用下去。所以不管是从愿不愿意用、还是从用不用得来这两头看，系统要推开都有不小的余地。

这么几条看下来，本系统贴着业务走、学起来用起来都不费劲、几个人配合也顺当，**操作可行性成立**。

## （四）本章小结

这一章是从技术、经济、操作这三头，分别给农村流水席信息管理系统的可行性作了论证。看下来大致是这样：技术上，挑的方案够成熟可靠，配套也算齐整，开发者也能胜任；经济上，钱花得少、效益却实在，农村小规模团队负担得起；操作上，贴着真实业务走，学和用都不难。这么算，本系统不管哪一头都站得住，可以接着往下，进到需求分析和系统设计的阶段了。
